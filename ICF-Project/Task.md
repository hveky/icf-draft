# Task

## 1. 当前项目任务定义

本项目当前的持续任务是：

1. 以标准模板为骨架，稳定生成训练集和测试集的修订记录文档
2. 以源文档推断结果作为输出内容，以验收文件作为离线金标准执行严格门禁
3. 明确区分 `draft` 草稿输出和 `release` 正式交付
4. 将规则、方法和失败模式沉淀到 skill 与文档中

## 2. 当前阶段性交付

### 2.1 已形成的长期能力

1. `draft / release` 双模式运行
2. 模板完整性校验
3. 严格交付校验
4. 诊断 alignment 报告
5. skill 与 Web 的统一运行壳层

### 2.2 当前必须长期保持的状态

1. `release` 不得通过验收行回放取得 `release_passed`
2. `release_blocked` 必须暴露字段、顺序、样式、缺失和额外行差异
3. 模板完整性校验必须持续通过
4. 任何新增规则都必须同时有测试和文档沉淀

## 3. 日常任务拆分规则

1. 单次任务若超过 3 个文件，必须拆分批次执行
2. 每个批次遵循：
   - 先说明方案
   - 再补测试
   - 再改实现
   - 再跑回归
3. 文档任务也遵循同样的拆分规则

## 4. 回归要求

每次涉及交付模型、模板写回、比较器、服务接口或 Web/skill 入口的改动后，应至少执行：

1. 目标测试
2. 全量 `pytest`
3. 训练集 `release` 回归
4. 测试集 `release` 回归
5. 检查以下工件：
   - `*.docx`
   - `*.report.json`
   - `*.acceptance_diff.json`
   - `output/progress/*.md`

## 5. 文档维护任务

当项目发生以下变化时，必须同步更新文档：

1. 正式交付门禁变化
2. 交付模型字段变化
3. 模板保护策略变化
4. skill 入口变化
5. 仓库运行方式变化

需要同步的文件包括：

1. `agent.md`
2. `Requirement.md`
3. `Design.md`
4. `Task.md`
5. `README.md`
6. skill 相关文档

## 6. 输出管理

1. 所有阶段成果统一进入 `output/`
2. 命名统一遵循 `R{轮次}-{节点名}`
3. 复盘统一进入 `output/progress/`
4. 不保留无法解释来源的临时交付件

## 7. 后续迭代方向

1. 继续收敛训练集与测试集的 source-to-delivery 规则，使严格交付在不依赖验收行回放的情况下也保持稳定
2. 继续压缩 `blocking_issues`
3. 在不破坏模板保护前提下，进一步提升章节/页码、原因措辞和表格片段选择的稳定性
4. 视需要再进入多模板适配，但当前不作为默认任务
